# 10. AIの作業タスク（メモの書き方）

AIの作業タスク（セクション11以降）は以下のテンプレートに沿って書く。各セクションが**単独で完結**していること（他セクションへの言及や「現〇〇の問題点」等のメタ議論を含めない）が重要。

---

## テンプレート

```markdown
# NN. AIの作業タスク X

**タスク**: （1行で端的に。例: 書類整合・docs-syncタグ打ち）

## 目的

なぜこのタスクをやるのか。AIが読んで「何のための作業か」を即座に理解できること。

## 現状

今の状態（何が整っていて、何が不足しているか）。

## 課題

放置した場合のリスク、または作業が必要な理由。

---

## 運用モード（必要なら）

タスクの性質によって手順が分岐する場合はここに書く。不要なら省略。

- **モードA: XXX** → 〇〇ルート
- **モードB: YYY** → 〇〇ルート

---

## Step 1. 〇〇

何をするか、具体的なコマンド・対象ファイル・判断基準を書く。

## Step 2. 〇〇

（以下同様）

## Step N-2. QCチェック

作業完了の検証手段。チェックリスト形式。

- 機械的チェック（grep・ファイル一覧比較などコマンドで検証可能なもの）
- 目視チェック（人間の判断が必要なもの）

## Step N-1. コミット & プッシュ

\`\`\`bash
git add -A
git commit -m "〇〇"
git push origin main
\`\`\`

## Step N. 仕上げ（タグ付与・履歴更新など）

タスク固有の後処理。

---

## 中断・失敗時の運用

- 途中で中断したらどうするか
- 想定外のエラー・発見への対処方針（本タスクで扱わず別タスクに切り出すなど）

---

**ステータス**: 着手予定 / 進行中 / 完了
```

---

## 書き方のルール

1. **単独完結**: 他セクションへの指摘や比較を書かない。AIがそのセクションだけ読めば実行できる状態にする
2. **目的・現状・課題を先頭に**: 「なぜやるか」を明示してからStepに入る
3. **Stepは番号順の命令形**: 各Stepは独立した実行可能単位。判断が必要なら基準を明示
4. **QCは後半に必ず含める**: 「終わったか否か」を確認できるチェックリストを用意
5. **コミット&プッシュ・タグ付け等の仕上げは最後のStep**: 書類変更・実装変更が伴う場合は必ず最後にgit操作を含める
6. **中断・失敗時の運用**: 長期タスク・多対象タスクは中断可能性を織り込む
7. **ステータスは末尾**: 着手予定 / 進行中 / 完了 のいずれか

# 11. AIの作業タスク A

**タスク**: UI/UXの美化とアニメーションの追加、スキン切り替えの設計

## 目的

ポーカーのテーブルをイメージしたリッチで美しいモダンなカジノ風UI（ダークテーマ、半透明ガラスモフィズムなど）を構築し、カード配布や正解・不正解時の3Dフリップ等のアニメーションを追加する。また、将来ガチャなどでトランプやテーブルのデザイン（スキン）を着せ替えられるように、CSS変数やJSONに基づくテーマ切り替えの基盤を実装する。

## 現状

- 簡易的なクイズUIが `src/App.svelte` に配置されている。
- スタイルは簡易的なものであり、カードの陰影やアニメーション、カジノ風のテーマは存在しない。
- スキン切り替えの仕組みはない。

## 課題

- 見た目がシンプルすぎて、ポーカーゲームとしての臨場感やワクワク感に欠ける。
- カードが切り替わる際に動きがないため、操作感が不自然である。
- 将来スキンを差し替えるための設計が整っていない。

---

## Step 1. スキン定義ファイルの作成
スキン切り替えの基盤として、テーマごとの配色やスタイル（フォント、背景、枠線、カードの影、スート色など）を定義する `src/data/skins.json` を作成する。

## Step 2. Svelteコンポーネントの作成
カード描画用コンポーネント `src/lib/components/Card.svelte` を新規作成する。
3Dフリップ効果やスート（マーク）の表示を担当する。

## Step 3. CSS変数を適用したスタイリング
`src/app.css` もしくは `src/App.svelte` にCSSカスタムプロパティ（CSS変数）を導入し、スキン定義に連動してUI（背景、ボタン、カード）全体のテーマが一括で切り替わるようにする。

## Step 4. 配布アニメーションの追加
新しいクイズが開始された際に、カードが画面下部や中央からスライドインしつつ3Dフリップ（裏から表にめくれる）するアニメーションをSvelteのトランジションやCSSキーフレームで実装する。

## Step N-2. QCチェック

- [ ] ローカルサーバーを起動し、カジノ風のダークテーマが美しく表示されることを確認する。
- [ ] クイズが進むたびにカードが滑らかにアニメーション（スライドイン＆フリップ）することを確認する。
- [ ] 将来スキンを切り替えるためのスキン定義 `src/data/skins.json` からスタイルが正しく読み込まれているかを確認する。

## Step N-1. コミット & プッシュ

```bash
git add -A
git commit -m "feature: UI/UXの美化とアニメーション追加、スキン切り替え設計の実装"
git push origin feature/rich-ui
```

## Step N. 仕上げ（タグ付与・履歴更新など）

- CHANGELOG.md を更新する。
- メモ帳（memo.md）のステータスを「完了」にする。

---

## 中断・失敗時の運用

- アニメーションのパフォーマンスが著しく悪い場合は、3D回転などを簡略化し、GPU負荷の少ないフェードイン・スライドに変更する。

---

**ステータス**: 完了
